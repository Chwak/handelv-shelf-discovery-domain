HANDLER: interpret-collector-query-lambda.ts (CommonJS exports.handler)

PRIMARY LOGIC
Authenticated users send naturalLanguageQuery. Empty input returns sanitized default JSON (null query, “No specific criteria”). Else builds prompt combining a fixed SYSTEM_PROMPT for Titan and user text, invokes BedrockRuntime amazon.titan-text-express-v1 with low temperature, extracts first JSON object from model output, sanitizeOutput clamps fields (query, categoryId, min/max price, materials array, summary) with length limits. On Bedrock failure logs error and falls back to echoing trimmed user text as query and summary. initTelemetryLogger is not called on success path before auth (only implicit via early returns).

PROS
- Graceful degradation to keyword fallback when model fails.
- Sanitization reduces oversized or malformed model output risk.

CONS
- LLM may hallucinate filters — clients should treat output as hints.
- Category vocabulary is prompt-constrained, not validated against real category IDs in code.
- Uses require() for AWS SDK and Bedrock — mixed module styles.

SECURITY
User-provided text sent to Bedrock — consider data residency and logging redaction; authenticated access only.

PERFORMANCE
Dominated by Bedrock InvokeModel latency; maxTokenCount 512.

FURTHER IMPROVEMENTS
Add telemetry init consistently; validate categoryId against enum from product service; circuit-break Bedrock when quota exceeded.

EXPOSES SENSITIVE INFORMATION
PARTIAL — user search phrases leave the account to the model provider; avoid logging full prompts at INFO in prod.

USER-FACING
YES — assistant feature in collector search UX.

EDGE CASES
PARTIAL — handles empty query, JSON parse failure (fallback), model empty output (fallback).
